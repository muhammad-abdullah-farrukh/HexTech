# Session 2: 2026-07-07 — pipeline resumability + graph correctness (duplicate nodes, hallucinated entities)

## What was wrong

1. **No way to resume `pipeline.py` from a specific stage.** All 9/10 stages'
   intermediate output only lived in Postgres (`pipeline_runs`,
   `graph_entities`/`graph_relationships`) — nothing on disk, so there was no
   way to inspect or hand-edit an intermediate artifact (e.g. the generated
   ontology) and re-run just the downstream stages without a full re-run.

2. **Duplicate graph nodes for the same real-world entity** — e.g.
   `scikit-learn` existed as two separate Neo4j nodes,
   `.../entity/scikit_learn` and `.../entity/Scikit_learn`, splitting its
   edges across two identities.

3. **Hallucinated entity substitution (severe).** Résumé education entries
   were replaced with completely wrong institutions in the graph:
   `"Air University"` → `"Brown University"`, `"Pakistan Intl. School"` (a
   Matric-level high school) → `"Birla Institute of Technology and Science,
   Pilani"` (an Indian university) — confirmed the corruption was introduced
   in Ontogen, not Phase 1 parsing (`04_structured.json` had the correct
   institution names).

## What we fixed

**1. Disk mirroring for every stage** (`ontogen/pipeline.py`):
   `outputs/stages/<resume_uuid>/stageN_*.json|.ttl` now gets written
   alongside every existing Postgres write (`_save_stage_to_disk`), purely
   additive — nothing reads these back on its own.

**2. `--from-stage=<stage>` resume mode** (`ontogen/pipeline.py`): loads every
   stage strictly before the given one from disk instead of recomputing it
   (`_STAGE_ORDER`, `_skip_before`, `_stage_value`, `_load_stage_from_disk`).
   Added a new disk-only artifact, `stage3_5_edc_canon_result.json` (the
   post-EDC `match_results`), since nothing previously persisted that
   intermediate value — needed to resume cleanly right before the `ontology`
   stage. Valid stage keys: `cq_gen, cq_answer, relation_extract,
   match_validate, edc_canon, ontology, stage9_10`. Must be single-token
   (`--from-stage=x`, not two args) — the CLI parser has no `argparse` and
   would otherwise swallow the value as the résumé UUID.

**3. Duplicate-node root cause #1 — `_slugify()` didn't case-fold**
   (`ontogen/db/kg_staging.py`): `"Scikit-learn"` and `"scikit-learn"` slugged
   to different URIs (`Scikit_learn` vs `scikit_learn`), so Neo4j's
   `MERGE (e:Entity {uri: ...})` treated them as different nodes. Fixed by
   lowercasing before slugifying — the one choke point every non-QID entity
   URI passes through.

**4. Duplicate-node root cause #2 — gazetteer lookups scoped too strictly by
   `entity_type`** (`ontogen/db/gazetteers.py`, `ontogen/stages/
   canonicalize.py`): the gazetteer only had `"skill"`-typed rows for
   `scikit-learn`, none `"technology"`-typed, so a mention classified as
   `"technology"` missed Tier 1/2 entirely and fell to Tier 3's free-form LLM
   generation (inconsistent casing). Added `lookup_any_type`,
   `canonical_values_any_type`, `get_qid_any_type` as fallbacks used by
   `_tier1`/`_tier2`/`_get_qid` when the type-scoped query misses.

**5. Hallucinated-entity root cause — Tier 2 (embedding) blindly accepted a
   cosine-similarity match ≥0.88 with no verification step**
   (`ontogen/stages/canonicalize.py`). Confirmed empirically against the live
   resolver + full gazetteer + production embedding model
   (`BAAI/bge-small-en`): `"Air University"` scored **0.884** against
   `"Brown University"` — just over the threshold — because generic shared
   words ("University") dominate short-string embeddings for entities with
   little other semantic content. This is the *entity* analogue of a bug
   `stage6_match_validate.py` already had to fix for *relationship* matching
   (see its `AUDIT NOTE` — same failure shape: embedding retrieves a
   plausible-but-wrong candidate, nothing verifies it before accepting).
   Fixed by mirroring stage6's retrieve-then-verify design:
   - `_content_tokens` / `_is_acronym_expansion` / `_passes_lexical_guard`: a
     deterministic guard rejecting a "match" that shares zero distinguishing
     words with the mention (exempting recognized acronym expansions like
     `MIT` → `Massachusetts Institute of Technology`).
   - `_llm_confirms_entity_match` (new): Tier 2 now requires an LLM to
     confirm a same-string/non-acronym candidate before accepting it —
     `_ENTITY_VERIFY_PROMPT`, `_ENTITY_ACCEPT_CONFIDENCE_THRESHOLD = 0.65`,
     same "yes 87"/"no 12" format stage6 already uses. Only fires for the one
     candidate that already cleared the cosine threshold — not per-entity.
   - Tier 3's own prompt (`_ENTITY_LLM_PROMPT`) rewritten to explicitly
     forbid substituting a different entity ("a wrong answer is worse than no
     answer"), instead of the old "normalize to most widely recognized form"
     phrasing that invited exactly this failure.
   - `_llm_confirms_entity_match`'s `call_llm` bumped to `max_tokens=600`
     (was 200) — this reasoning model burns unpredictable budget on
     `<think>` reasoning before the real answer; 200 was truncating to empty
     content, which happened to fail *safe* (empty → parsed as "no" → falls
     through to Tier 3) but silently defeated Tier 2 on every borderline
     match.

## What went wrong (during the fix)

- **First diagnosis of the hallucination was wrong.** Initially assumed Tier
  3 (LLM) was the culprit based on its "normalize to most widely recognized
  form" prompt looking suspicious, and shipped a prompt rewrite + lexical
  guard for Tier 3 alone. Live-testing against the real resolver afterward
  showed the fix didn't work — the entity never reached Tier 3 at all,
  because Tier 2 was accepting it first. An earlier manual embedding test
  that seemed to rule out Tier 2 (`0.71` similarity, safely under 0.88) used
  a smaller hand-picked candidate list and a slightly different embedding
  model name (`bge-small-en-v1.5` vs. the production `bge-small-en`) — against
  the real 147-entry gazetteer and the real model, the score was `0.88+`.
  Lesson: when root-causing a resolver/matching bug, reproduce against the
  *exact* production model name and the *full* real candidate set, not a
  representative subset — a "safely below threshold" result on a partial
  repro is not evidence the real path is safe.
- Repeated pipeline re-runs during unrelated debugging (checking whether the
  process was "stuck") accumulated duplicate `graph_entities`/
  `graph_relationships` rows for the same résumé, since `kg_staging.py`'s
  writers are plain `INSERT`s with no upsert/dedup — had to be manually
  cleared (`DELETE ... WHERE source_doc = ...`) before re-staging.
- `load_to_neo4j.py --wipe` clears Neo4j but does **not** reset Postgres's
  `synced_to_neo4j` flags — after a manual `cypher-shell ... DETACH DELETE`
  wipe, reloading only pushes rows still marked unsynced, silently dropping
  any résumé whose rows were already marked synced before the wipe. Had to
  scope cleanup to one résumé at a time (`DELETE ... WHERE source_doc = ...`
  in Postgres + a targeted `DETACH DELETE` of just the known-bad Neo4j nodes)
  rather than a full wipe, to avoid losing the other two résumés' correct
  data.

# Session 3: 2026-07-08 — regressions from Session 2's fixes (case-duplicate nodes, corrupted names, project mis-canonicalization)

## What regressed (found by post-rebuild graph audit)

Verification of the rebuilt Neo4j graph after Session 2's fixes showed 144
nodes but only 87 case-insensitively-distinct URIs — **57 case-duplicate
nodes** (`wd:Python` vs `wd:python`; *two Abdullah Zahid person nodes* with
divergent edges — the old one missing the education fixes entirely), plus
**identity corruption**: the node with URI `wd:Machine_Learning` had
`name: "Predictive Optimization Models"`, `wd:Deep_Learning` had
`name: "Reinforcement Learning Suite"`, and the person had `HAS_PROJECT`
edges pointing at those skill-concept nodes.

## Root causes

1. **Session 2's slugify fix covered only 1 of THREE URI-minting sites.**
   `db/kg_staging.py` (Path A) was lowercased, but Path B mints `wd:` URIs
   through two other, still case-preserving slugifiers:
   `stages/stage9_10_kg.py` `_slugify` (used by `facts_to_graph` for
   subject/object URIs) and `stages/canonicalize.py` `_slugify` (used by
   `resolve_kg_entities` when rewriting URIs to canonical forms). Path A
   emitted lowercase URIs, Path B mixed-case → the same entity split into two
   Neo4j nodes per path. This split even the person node.

2. **`resolve_kg_entities` rewrote URIs but never the `rdfs:label`.** A
   project URI rewritten to a canonical URI carried the old project label;
   `stage_graph` maps `rdfs:label` → node `name`, and `load_to_neo4j`'s
   `SET e += row` overwrites last-write-wins — so the legitimate
   `Machine Learning` node got renamed to the project.

3. **Project names were canonicalized against the skills gazetteer.**
   `PROPERTY_ENTITY_TYPE_MAP` had no project predicate → `HasProject` objects
   got type `"unknown"` → Session 2's `canonical_values_any_type` fallback
   compared them against ALL gazetteer canonicals. "Predictive Optimization
   Models"→"Machine Learning" passed **Tier 2** (cosine ≥0.88 AND the LLM
   confirmation said yes); "Reinforcement Learning Suite"→"Deep Learning"
   passed **Tier 3** (lexical guard only rejects fully-disjoint token sets;
   they share "learning").

## Fixes applied

- Lowercased `stage9_10_kg._slugify` and `canonicalize._slugify`; aligned the
  empty-name sentinel to `"unknown"` across all three slugifiers. URIs are
  merge keys; display casing lives in `rdfs:label`/node `name`.
- `resolve_kg_entities`: URI-collection filter changed from the "has
  underscore or starts uppercase" heuristic (which would skip lowercase
  single-word slugs like `python`) to an explicit QID exclusion
  (`^Q\d+$`); and the rewrite loop now rewrites the `rdfs:label` to
  `canonical_form` whenever it rewrites the URI.
- `ResumeEntityResolver.resolve`: hard gate — types in
  `{"project", "person", "unknown"}` are never canonicalized (tier
  `"unresolved"`, mention returned unchanged). Covers both Path A and Path B
  callers.
- The any-type gazetteer fallbacks (`lookup_any_type` /
  `canonical_values_any_type` / `get_qid_any_type`) now only fire for
  `{"skill", "technology", "tool", "library", "framework"}` — preserves the
  scikit-learn fix, stops cross-type matches for
  company/university/certification/job_title.
- `PROPERTY_ENTITY_TYPE_MAP` += `HasProject`/`Project`/`BuiltProject` →
  `"project"`.
- Tier 3 now persists accepted, changed normalizations via the
  previously-never-called `gaz_db.add_alias` (source `tier3_llm`,
  confidence ≥ 0.7) so repeat mentions resolve deterministically via Tier 1.
- New regression tests: `tests/test_entity_resolution_regressions.py`
  (slugifier parity/lowercasing, the type gate, label-follows-URI). Stale
  expectation in `test_ontogen_db.py::test_stage_structured_relations`
  updated (`wd:Jane_Doe` → `wd:jane_doe`). 16/16 pass. Live smoke: project/
  unknown gated; "Air University" unchanged; scikit-learn-as-technology now
  Tier 1.

## Lessons

- **When a fix targets a duplicated helper, grep for every copy first.**
  `_slugify` existed in three files; fixing one guaranteed the split.
- **A URI rewrite must move the label with it** — any identity-changing
  rewrite that leaves display properties behind corrupts the merge target.
- **An "everything" fallback needs a type gate.** The any-type fallback was
  correct for tech terms and catastrophic for résumé-unique types.

## Outstanding

- The rebuild (user runs it) — until then Neo4j/Postgres still hold the
  corrupted mixed-scheme data.
- Fabricated stage-1–8 skills (hallucinated during extraction, cached in
  pipeline_runs) are NOT addressed by this rebuild — separate task: clear
  pipeline_runs for affected résumés + full non-cached rerun + add a
  grounding check to extraction.
- Distinct-mention pairs (`Deep Q-Networks`/`Deep Q-Network`,
  `Shocks & Stars Engineering`/`…Ltd.`) may still split; alias-learning
  reduces recurrence but can't guarantee semantic merges.

# Session 4: 2026-07-08 — LLM concurrency attempt (reverted), hasSkill retry-storm fix, Path A/Stage 6 waste removed, model swap

## Concurrency attempt (tried, reverted)

Restarted `llama-server` with `-np 3 -c 24576 -fa on -ctk/-ctv q8_0` (3 parallel
inference slots) to run 3 résumés' `pipeline.py` concurrently in separate
terminals. Two blockers found, in order:

1. **`resume_parser/llm_lock.py`** is a strict single-holder file lock shared
   between the parser and Ontogen (`"Both the parser and Ontogen call one
   llama-server running --parallel 1"`) — a 2nd concurrent `pipeline.py`
   process aborted immediately with `LLM appears busy`, regardless of the
   server's actual slot count. Rewrote it as a counting semaphore
   (`$HEXTECH_LLM_SLOTS`, default 3) with an `flock`-guarded mutex file
   around every read-modify-write (closing the TOCTOU race a naive counting
   lock would have) and atomic tmp-file + `os.replace` writes. Verified live
   with a real multi-process race test (not just read-through): exactly 3 of
   5 racing processes acquired, a `SIGKILL`ed holder's slot was correctly
   pruned and reclaimed. `REQUEST_TIMEOUT` was also bumped 300→900 for the
   GPU-sharing headroom.
2. **GPU VRAM exhaustion**, independent of the lock: `-np 3 -c 24576
   -ctk/ctv q8_0` on the 32B model alone consumes ~23.7/24.5 GB VRAM
   (`nvidia-smi`), leaving ~320 MB free. `SentenceTransformer(EMBED_MODEL)`
   (`stage6_match_validate.py`, `canonicalize.py` — 3 call sites) defaults to
   CUDA with no device override, so it hit `CUDA error: out of memory` on
   load. This would fail even running résumés one at a time under this server
   config, not just under concurrency — the fix (pin `EMBED_DEVICE=cpu`,
   `bge-small-en` is small enough CPU is not a bottleneck) was scoped but
   **not applied** — user chose to abandon the concurrency attempt instead.

**Both `llm_lock.py` and the `REQUEST_TIMEOUT` bump were reverted** (`git
checkout --`) back to the single-holder lock / 300s timeout, and the server
was reverted to a single-slot config. If concurrency is revisited later: the
counting-lock design is sound (was independently reviewed by another model
and matched this session's own race-condition analysis) but was never
re-applied; the `EMBED_DEVICE=cpu` fix is a prerequisite, not optional,
under any multi-slot server config with `-ngl 99`.

## Bugs found and fixed on real data (kept)

**1. Stage 7/8 `hasSkill` retry-storm** (`stages/stage7_8_ontology.py`,
`build_ontology`). `seen_new_props` only registered a property as "handled"
*on success* (after passing Turtle validation). Path A emits one `hasSkill`
relation instance *per skill* (Zahid: 41 skills → 41 instances), and the LLM
was deterministically generating the same malformed Turtle for `hasSkill`
every time (identical 7663-char response, identical line-154 syntax error) —
so every one of the 41 instances re-triggered a fresh, doomed
`_new_prop_turtle()` LLM call, since the failing key never got recorded.
Fixed by adding `attempted_new_props: set[str]`, populated *before* calling
`_new_prop_turtle()` regardless of outcome — each unique property name now
gets exactly one LLM attempt per ontology build, success or fail. Protects
any property with multiple instances, not just Path A ones.

**2. `pipeline.py:309` — Path A relations no longer feed Stage 6/EDC/ontology.**
`relations = list(cq_rels) + list(struct_rels)` merged Path A's structured
relations (fixed schema: `email`, `phone`, `employer`, `jobTitle`, `startDate`,
`endDate`, `educatedAt`, `degree`, `graduationYear`, `hasSkill`, `hasProject`,
`usesTechnology`) into the pool re-validated against Wikidata (Stage 6) and
re-defined via EDC every single run, for every résumé — pure waste, traced
end-to-end: `kg_staging.stage_structured_relations()` (Path A's actual graph
write) never reads `match_results`/`ontology` at all; it maps
`rel["property"]` straight to a relationship type via a static regex
(`_rel_type`) and resolves entity values via `ResumeEntityResolver`
independently. Measured on Zahid's résumé: 13 of Stage 6's 25 unique
properties were Path A's fixed set, roughly half his runtime. Changed to
`relations = list(cq_rels)` — Stage 6 confirmed dropping from
"111 relations → 25 unique properties" to "14 relations → 13 unique
properties" on the next run. Soft, non-correctness edge case accepted: if
free-text project descriptions ever restate a Path-A-captured fact, Path B
may mint a differently-named edge for it instead of being pre-aligned via the
shared EDC pass — graph-modeling redundancy, not wrong data, and already
possible today regardless of this change.

**3. Model swap**: `config.py`'s `LLM_MODEL`/`LLM_BASE_URL` switched from the
local llama-server DeepSeek-R1-Distill-Qwen-32B GGUF to a vLLM endpoint
serving `/app/local_model/Qwen3.5-9B` (same port, `127.0.0.1:9000`,
`max_model_len` 8192 confirmed via `/v1/models`). Comment above `LLM_MODEL`
updated accordingly (was documenting the old 24576-ctx llama-server setup).

## Lessons

- **A per-run dedup set that's only populated on success will retry forever
  on a deterministic failure**, once for every instance of the failing key —
  always mark "attempted" before the risky call, not just "succeeded" after.
- **Trace whether a pipeline stage's output is actually consumed before
  assuming it's needed.** Stage 6/EDC/ontology looked load-bearing for Path A
  because it ran on Path A's data, but Path A's own persistence never reads
  any of those stages' output — the merge was pure legacy cost.
- **A file lock's stated invariant can go stale** when unrelated
  infrastructure changes (server `-np` count) without the lock being
  revisited — `llm_lock.py`'s docstring literally said `--parallel 1`.
- Two independent VRAM consumers on one GPU (`llama-server` with `-ngl 99`
  full offload, and any Python-side `SentenceTransformer`/torch model) can
  each individually assume the whole card and starve the other — check
  `nvidia-smi` free memory before raising a server's context/parallelism
  flags, not just after something OOMs.

  future multi-slot (`-np > 1`) server config is used again.
- The counting-lock `llm_lock.py` rewrite — reverted, not currently in the
  codebase; would need to be reapplied (it's not saved anywhere but this
  log's description of the design) if concurrency is revisited.
- Farrukh's and Riyan's résumés still not rebuilt/verified as of this
  session — Zahid's rebuild (with Session 3 + Session 4's fixes) was
